Uvod: Šta su OWIN i ASP.NET Identity?

Prije nego počnemo s implementacijom, važno je razumjeti zašto ove tehnologije postoje i šta su zamijenile. Ovaj uvod objašnjava historiju, svrhu i arhitekturu obje komponente.

Šta je postojalo prije?

SqlMembershipProvider (ASP.NET 2.0, 2005)
Prva "ugrađena" podrška za autentifikaciju u ASP.NET-u. Radila je uz aspnet_regsql.exe koji je kreirao fiksnu šemu tabela (aspnet_Users, aspnet_Membership, aspnet_Roles...) direktno u SQL Server.

Problemi:
  • Nije se moglo proširiti (dodati npr. FullName, DepartmentID) bez zasebnih tabela i ručnog spajanja
  • Čvrsto vezano za SQL Server — nije radilo s MySQL, PostgreSQL, Azure Table Storage
  • Lozinke hashirane slabim algoritmom (MD5 ili SHA-1 bez soli)
  • Tightly coupled s IIS i System.Web — nije radilo van web konteksta (unit testi, konzolne aplikacije)
  • Nije podržavalo Claims — moderan pristup autentifikaciji
SimpleMembership / WebMatrix (2011)
Pokušaj pojednostavljivanja — novi API koji je dopuštao prilagodbu šeme baze. Ipak, još uvijek vezan za System.Web, složen za testiranje, i bez Claims podrške. Nije postao standard jer je MVC 4 / WebAPI tim već radio na nečem boljem.

Šta je ASP.NET Identity?

ASP.NET Identity je sistem za upravljanje korisnicima koji je Microsoft izdao 2013. zajedno s ASP.NET MVC 5. Zamijenio je SqlMembershipProvider i SimpleMembership.

Ključni dizajnerski ciljevi:

  • Proširivost — IdentityUser je obična C# klasa; dodaješ polja kao što bi dodao u bilo koji model
  • Neovisnost o bazi — "Store" pattern razdvaja logiku od perzistencije. Postoje EF store, Dapper store, Azure store... možeš pisati vlastiti
  • Testabilnost — UserManager<T> radi bez IIS-a; može se unit testirati s mock store-om
  • Claims-based identitet — svaki korisnik = ClaimsIdentity s kolekcijom key-value parova (claim-ova). Uloga "Admin" je samo claim tipa ClaimTypes.Role
  • Sigurniji password hashing — PBKDF2 s 1000 iteracija i SHA-1 (Identity v2); Identity v3 (.NET Core) koristi SHA-256
Arhitektura ASP.NET Identity 2.x
// Sloj 1: MODELI — definiraju šemu korisnika
IdentityUser               // baza: Id, UserName, Email, PasswordHash, SecurityStamp...
  └── ApplicationUser      // naša proširena klasa: + FullName, DepartmentID

IdentityRole               // baza: Id, Name

// Sloj 2: STORES — perzistencija (čitanje/pisanje u bazu)
IUserStore<TUser>          // sučelje koje UserManager poziva
  └── UserStore<TUser>    // EF implementacija (Microsoft.AspNet.Identity.EntityFramework)

IRoleStore<TRole>         // sučelje
  └── RoleStore<TRole>   // EF implementacija

// Sloj 3: MANAGERS — poslovna logika (validacija, hashiranje, lockout...)
UserManager<TUser>        // CreateAsync, FindByEmailAsync, AddToRoleAsync, CheckPasswordAsync...
  └── ApplicationUserManager  // naša konfiguracija (password pravila, lockout)

RoleManager<TRole>        // RoleExistsAsync, CreateAsync...
  └── ApplicationRoleManager

SignInManager<TUser, TKey> // PasswordSignInAsync — provjerava lozinku + izdaje cookie
  └── ApplicationSignInManager

// Sloj 4: BAZA — EF Code-First kreira ove tabele automatski
AspNetUsers          // Id, UserName, Email, PasswordHash, SecurityStamp, FullName, DepartmentID
AspNetRoles          // Id, Name
AspNetUserRoles      // UserId, RoleId (many-to-many veza)
AspNetUserClaims     // UserId, ClaimType, ClaimValue
AspNetUserLogins     // UserId, LoginProvider, ProviderKey (za OAuth/Google login)
Šta su Claims?
Claim je jednostavan key-value par koji opisuje korisnika: { Type: "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name", Value: "admin@demo.com" }
{ Type: "http://schemas.microsoft.com/ws/2008/06/identity/claims/role", Value: "Admin" }

User.IsInRole("Admin") interno traži claim tipa ClaimTypes.Role s vrijednošću "Admin".
User.Identity.Name vraća claim tipa ClaimTypes.Name.

Prednost claims-based pristupa: sve informacije o korisniku putuju u šifriranom cookie-ju — svaki zahtjev ne mora ići u bazu kako bi se provjerila uloga.

Šta je OWIN?

OWIN (Open Web Interface for .NET) nije biblioteka — to je specifikacija (standard) koja definira kako web server i web aplikacija trebaju komunicirati, bez da su čvrsto vezani jedan za drugog.

Klasični ASP.NET MVC je čvrsto vezan za System.Web.dll i IIS pipeline. Svaki zahtjev prolazi kroz desetine IIS HttpModules (autentifikacija, session, kompresija...) čak i za API pozive koji ništa od toga ne trebaju. Nije se moglo pokrenuti van IIS-a.

OWIN specifikacija (2012) definira jedno jednostavno sučelje: 

// OWIN aplikacija je samo jedna funkcija:
using AppFunc = Func<
    IDictionary<string, object>,  // environment: request + response podaci
    Task                           // asinkrono, bez povratne vrijednosti
>;

// Middleware je "omotač" oko te funkcije:
// prima zahtjev, može ga promijeniti, prosljeđuje sljedećem middleware-u, obradi odgovor

Microsoftova implementacija OWIN specifikacije zove se Katana (2013). Uz Katana dolaze paketi:

  • Microsoft.Owin — osnovna implementacija, IOwinContext, IOwinRequest, IOwinResponse
  • Microsoft.Owin.Host.SystemWeb — adapter koji ubacuje OWIN pipeline unutar klasičnog IIS pipeline-a (bridge)
  • Microsoft.Owin.Security.Cookies — middleware za cookie autentifikaciju

Zašto je OWIN bio revolucionaran za klasični ASP.NET?
Omogućio je da autentifikacijski kod radi neovisno od IIS-a — isti UseCookieAuthentication middleware može raditi na IIS-u, na OWIN self-host konzolnoj aplikaciji, ili u Nancy/ServiceStack frameworku — jer ovisi samo o OWIN specifikaciji, ne o System.Web.

Ovo je i razlog zašto ASP.NET Core nije trebao OWIN kao poseban sloj — Core je od početka dizajniran s ugrađenim middleware pipeline-om (isti koncept, ali nativni).

Kako OWIN i ASP.NET Identity rade zajedno?

ASP.NET Identity nije vezano za OWIN — UserManager i RoleManager rade bez OWIN-a (mogu se koristiti u konzolnoj aplikaciji). OWIN je potreban samo za isporuku autentifikacije kroz cookie middleware.

KomponentaOdgovornostPaket
UserManager Logika korisnika: kreiranje, provjera lozinke, uloge, lockout Identity.Core
SignInManager Orkestrira: provjeri lozinku → generiraj Claims → predaj OWIN-u da postavi cookie Identity.Owin
CookieAuthenticationMiddleware Na svakom zahtjevu: čita cookie → deserijalizira Claims → popuni User.Identity Owin.Security.Cookies
ApplicationDbContext EF Code-First perzistencija: čuva i čita Identity tabele iz SQL Server-a Identity.EntityFramework

Životni ciklus prijave (korak-po-korak):

  1. Korisnik unese email + lozinku, klikne "Prijava"
  2. AccountController.Login(POST) poziva SignInManager.PasswordSignInAsync(...)
  3. SignInManager poziva UserManager.FindByNameAsync(email) → vraća ApplicationUser
  4. UserManager.CheckPasswordAsync(user, password) → uspoređuje PBKDF2 hash
  5. Na uspjehu: user.GenerateUserIdentityAsync(manager) kreira ClaimsIdentity (s ulogama kao claim-ovima)
  6. OWIN IAuthenticationManager.SignIn(claimsIdentity) → šifrira identity → kreira Set-Cookie header
  7. Svaki naredni zahtjev: CookieAuthenticationMiddleware čita cookie → dešifrira → kreira ClaimsPrincipal → Thread.CurrentPrincipal = User = principal
  8. User.IsInRole("Admin") i [Authorize(Roles="Admin")] rade bez upita u bazu
Security Stamp — zašto postoji?
Cookie može živjeti satima ili danima. Šta ako SuperAdmin u međuvremenu obriše korisnikov nalog ili promijeni lozinku? Cookie je još uvijek validan!

SecurityStamp je GUID koji se mijenja svaki put kad se promijeni lozinka, uloga ili sigurnosni podatak. SecurityStampValidator u Startup.cs svakih 30 minuta provjerava u bazi: je li security stamp u cookie-ju isti kao u bazi? Ako nije — cookie se poništava i korisnik mora ponovo da se prijavi.

Ovo je razlog zašto Identity dodaje SecurityStamp kolonu u AspNetUsers.

Korak 2: Identity modeli

Nova datoteka: Models/IdentityModels.cs 
using Microsoft.AspNet.Identity;
using Microsoft.AspNet.Identity.EntityFramework;
using System.Security.Claims;
using System.Threading.Tasks;

namespace BHAS.Models
{
    // ApplicationUser nasljeđuje IdentityUser.
    // IdentityUser već ima: Id, UserName, Email, PasswordHash, SecurityStamp...
    // Mi dodajemo dva polja:
    public class ApplicationUser : IdentityUser
    {
        public string FullName { get; set; }   // za prikaz u navbaru
        public int? DepartmentID { get; set; } // za department-scoped pristup (Editor uloga)

        // Kreira ClaimsIdentity koji se čuva u auth cookie-ju.
        // Poziva ga OWIN pri svakom uspješnom loginu.
        public async Task<ClaimsIdentity> GenerateUserIdentityAsync(
            UserManager<ApplicationUser> manager)
        {
            var userIdentity = await manager.CreateIdentityAsync(
                this, DefaultAuthenticationTypes.ApplicationCookie);
            return userIdentity;
        }
    }

    // POSEBAN DbContext samo za Identity tabele.
    // Naslijeđuje IdentityDbContext koji automatski definira:
    //   AspNetUsers, AspNetRoles, AspNetUserRoles, AspNetUserClaims, AspNetUserLogins
    // Koristi "AuthDbConnection" = StateStatisticsAuthDB (odvojena baza!).
    public class ApplicationDbContext : IdentityDbContext<ApplicationUser>
    {
        public ApplicationDbContext()
            : base("AuthDbConnection", throwIfV1Schema: false) { }

        public static ApplicationDbContext Create() => new ApplicationDbContext();
    }
}
Zašto odvojena baza?
Aplikacija već koristi StateStatisticsDBEntities (DB-First EF) za StateStatisticsDB. Ako bismo koristili isti DbContext, EF Code-First bi pokušao upravljati šemom postojeće baze — što može uzrokovati brisanje tabela!

Rješenje: ApplicationDbContext koristi drugu bazu StateStatisticsAuthDB — EF je kreira automatski pri prvom pokretanju. U produkciji česta praksa je imati sve u jednoj bazi ali s odvojenim migracijama.
Dva konteksta — dva connection stringa:
StateStatisticsDBEntities → StateStatisticsDB (DB-First, EntityClient format)
ApplicationDbContext → StateStatisticsAuthDB (Code-First, SqlClient format)

Nikad ne miješaj instance ova dva konteksta — svaki radi s drugom bazom!
Dodaj u Web.config (u <connectionStrings>): 
<add name="AuthDbConnection"
     connectionString="Data Source=localhost,2433;Initial Catalog=StateStatisticsAuthDB;
       User Id=sa;Password=NasaLozinka!2026;TrustServerCertificate=True;
       MultipleActiveResultSets=True"

Korak 3: OWIN Startup i IdentityConfig

OWIN u klasičnom ASP.NET — kako se uklapa?
Paket Microsoft.Owin.Host.SystemWeb registrira OWIN pipeline kao HttpModule unutar klasičnog IIS pipeline-a. Konkretno: pri startanju aplikacije, ovaj paket traži klasu označenu s [assembly: OwinStartup(typeof(...))] i poziva njenu Configuration(IAppBuilder app) metodu.

IAppBuilder je centralni objekt OWIN konfiguracije — svaki poziv app.Use...() dodaje middleware u pipeline. Redoslijed registracije je važan: middleware se izvršavaju redom kojim su dodani.
Nova datoteka: Startup.cs (u korijenu projekta) 
using Microsoft.Owin;
using Owin;
using System;
using Microsoft.Owin.Security.Cookies;
using Microsoft.AspNet.Identity;
using Microsoft.AspNet.Identity.Owin;
using BHAS.Models;

// Ovaj atribut kaže OWIN-u: "kada aplikacija krene, pozovi Startup.Configuration()"
[assembly: OwinStartup(typeof(BHAS.Startup))]

namespace BHAS
{
    public class Startup
    {
        public void Configuration(IAppBuilder app)
        {
            // Registriraj factory metode za OWIN context.
            // Svaka factory se poziva jednom po HTTP zahtjevu — kreira novu instancu
            // i automatski je dispose-uje na kraju zahtjeva.
            app.CreatePerOwinContext(ApplicationDbContext.Create);
            app.CreatePerOwinContext<ApplicationUserManager>(ApplicationUserManager.Create);
            app.CreatePerOwinContext<ApplicationSignInManager>(ApplicationSignInManager.Create);
            app.CreatePerOwinContext<ApplicationRoleManager>(ApplicationRoleManager.Create);

            // Cookie autentifikacijski middleware.
            // Na svakom zahtjevu: čita auth cookie, validira ga, popunjava User.Identity.
            app.UseCookieAuthentication(new CookieAuthenticationOptions
            {
                AuthenticationType = DefaultAuthenticationTypes.ApplicationCookie,
                LoginPath = new PathString("/Account/Login"), // redirect za neautentificirane
                Provider = new CookieAuthenticationProvider
                {
                    // Svakih 30 min provjerava bazu: je li korisnik obrisan/promijenio lozinku?
                    OnValidateIdentity = SecurityStampValidator
                        .OnValidateIdentity<ApplicationUserManager, ApplicationUser>(
                            validateInterval: TimeSpan.FromMinutes(30),
                            regenerateIdentity: (mgr, user) =>
                                user.GenerateUserIdentityAsync(mgr))
                }
            });
        }
    }
}
Nova datoteka: App_Start/IdentityConfig.cs — definira ApplicationUserManager, ApplicationSignInManager i ApplicationRoleManager: 
// ApplicationUserManager — konfigurira password pravila, lockout, token provider
public class ApplicationUserManager : UserManager<ApplicationUser>
{
    public ApplicationUserManager(IUserStore<ApplicationUser> store) : base(store) { }

    public static ApplicationUserManager Create(
        IdentityFactoryOptions<ApplicationUserManager> options, IOwinContext context)
    {
        var manager = new ApplicationUserManager(
            new UserStore<ApplicationUser>(context.Get<ApplicationDbContext>()));

        // Pravila za username i email
        manager.UserValidator = new UserValidator<ApplicationUser>(manager)
        {
            AllowOnlyAlphanumericUserNames = false,
            RequireUniqueEmail = true
        };

        // Password pravila (opuštena za demo)
        manager.PasswordValidator = new PasswordValidator
        {
            RequiredLength = 6,
            RequireDigit = true,
            RequireLowercase = true,
            RequireNonLetterOrDigit = false,
            RequireUppercase = false
        };

        // Lockout: 5 neuspješnih pokušaja → zaključan 5 minuta
        manager.UserLockoutEnabledByDefault = true;
        manager.MaxFailedAccessAttemptsBeforeLockout = 5;
        manager.DefaultAccountLockoutTimeSpan = TimeSpan.FromMinutes(5);

        return manager;
    }
}

// ApplicationSignInManager — provjerava lozinku i izdaje cookie
public class ApplicationSignInManager : SignInManager<ApplicationUser, string>
{
    public ApplicationSignInManager(ApplicationUserManager mgr, IAuthenticationManager auth)
        : base(mgr, auth) { }

    public static ApplicationSignInManager Create(
        IdentityFactoryOptions<ApplicationSignInManager> options, IOwinContext context)
        => new ApplicationSignInManager(
            context.GetUserManager<ApplicationUserManager>(), context.Authentication);
}

// ApplicationRoleManager — kreira/čita uloge
public class ApplicationRoleManager : RoleManager<IdentityRole>
{
    public ApplicationRoleManager(IRoleStore<IdentityRole, string> store) : base(store) { }

    public static ApplicationRoleManager Create(
        IdentityFactoryOptions<ApplicationRoleManager> options, IOwinContext context)
        => new ApplicationRoleManager(new RoleStore<IdentityRole>(context.Get<ApplicationDbContext>()));
}
Šta radi CreatePerOwinContext<T>?
Ovo je OWIN-ov ekvivalent dependency injection registracije. Svaki poziv app.CreatePerOwinContext<T>(factory) kaže OWIN-u:
"Kada neko prvi put zatraži instancu T unutar ovog HTTP zahtjeva, pozovi factory, vrati tu instancu, i dispose-uj je kada se zahtjev završi."

Životni ciklus po zahtjevu:
  1. HTTP zahtjev stiže → OWIN kreira novi IOwinContext za ovaj zahtjev
  2. Kontroler poziva HttpContext.GetOwinContext().Get<ApplicationDbContext>()
  3. OWIN vidi da instanca još ne postoji za ovaj zahtjev → poziva ApplicationDbContext.Create()
  4. Ista instanca se vraća svim pozivima Get<ApplicationDbContext>() u istom zahtjevu
  5. Zahtjev se završava → OWIN automatski poziva Dispose() na svim instancama

Ovo garantira: jedan DbContext po zahtjevu (efikasno), automatsko zatvaranje konekcije (bez memory leakova).

U .NET Core ovo radi ugrađeni DI kontejner s AddScoped<T>: jedan scope = jedan HTTP zahtjev; konstruktorska injekcija umjesto ručnog GetOwinContext().GetUserManager<T>().
Zašto postoje tri manager klase?
ApplicationUserManager — radi s korisnicima (CRUD, lozinke, lockout, token generacija)
ApplicationSignInManager — orkestrira prijavu: provjeri lozinku + postavi cookie
ApplicationRoleManager — radi s ulogama (kreiranje uloga, provjera postoji li uloga)

Identity razdvaja ove odgovornosti jer:
  • Možeš imati UserManager bez SignInManager (npr. u pozadinskim servisima)
  • Možeš imati RoleManager bez UserManager
  • Svaki može imati svoju konfiguraciju, mock implementaciju u testovima, itd.

Korak 4: Automatsko seedovanje baze

Nova datoteka: Infrastructure/AuthDbSeeder.cs 
public static class AuthDbSeeder
{
    public static void Seed()
    {
        using (var db = new ApplicationDbContext())
        {
            // Kreira bazu StateStatisticsAuthDB ako ne postoji
            // (i sve AspNet* tabele unutar nje)
            db.Database.CreateIfNotExists();

            var roleManager = new RoleManager<IdentityRole>(new RoleStore<IdentityRole>(db));
            var userManager = new UserManager<ApplicationUser>(new UserStore<ApplicationUser>(db));

            // Kreira 4 uloge (ako ne postoje)
            string[] roles = { "SuperAdmin", "Admin", "Editor", "User" };
            foreach (var role in roles)
                if (!roleManager.RoleExists(role))
                    roleManager.Create(new IdentityRole(role));

            // Kreira SuperAdmin korisnika (ako ne postoji)
            if (userManager.FindByEmail("admin@demo.com") == null)
            {
                var superAdmin = new ApplicationUser
                {
                    UserName = "admin@demo.com",
                    Email = "admin@demo.com",
                    FullName = "Super Administrator",
                    EmailConfirmed = true
                };
                var result = userManager.Create(superAdmin, "Admin@123!");
                if (result.Succeeded)
                    userManager.AddToRole(superAdmin.Id, "SuperAdmin");
            }
        }
    }
}
Pozovi iz Global.asax.cs u Application_Start():

Kada se Seed() poziva?
Application_Start se izvršava jednom — kada IIS/IIS Express prvi put pokrene aplikaciju (ili nakon recycle-a). Seed metoda provjeri postoje li već uloge i korisnik, pa je sigurno pozivati je svaki put — neće kreirati duplikate.

U .NET Core seedovanje se radi u Program.cs odmah nakon app.Build(), ili kroz EF migration seed metode.
Pristup seedovanom nalogu:
Email: admin@demo.com  |  Lozinka: Admin@123!
Odmah nakon prve prijave, SuperAdmin treba kreirati ostale korisnike putem Upravljanje korisnicima.

Korak 5: AccountController — prijava i odjava

Nova datoteka: Controllers/AccountController.cs 
[Authorize]  // svi zahtjevi zahtijevaju login, osim onih s [AllowAnonymous]
public class AccountController : Controller
{
    private ApplicationSignInManager SignInManager
        => HttpContext.GetOwinContext().Get<ApplicationSignInManager>();

    // GET: Account/Login
    [AllowAnonymous]  // PREGAZI class-level [Authorize]
    public ActionResult Login(string returnUrl)
    {
        ViewBag.ReturnUrl = returnUrl;
        return View();
    }

    // POST: Account/Login
    [HttpPost, AllowAnonymous, ValidateAntiForgeryToken]
    public async Task<ActionResult> Login(LoginViewModel model, string returnUrl)
    {
        if (!ModelState.IsValid) return View(model);

        // PasswordSignInAsync: provjerava lozinku, prati failed attempts, izdaje cookie
        var result = await SignInManager.PasswordSignInAsync(
            model.Email, model.Password, model.RememberMe, shouldLockout: true);

        switch (result)
        {
            case SignInStatus.Success:
                return RedirectToLocal(returnUrl);
            case SignInStatus.LockedOut:
                ModelState.AddModelError("", "Nalog zaključan. Pokušaj za 5 min.");
                return View(model);
            default:
                ModelState.AddModelError("", "Pogrešan email ili lozinka.");
                return View(model);
        }
    }

    // POST: Account/Logout — mora biti POST (ne GET) iz CSRF razloga
    [HttpPost, ValidateAntiForgeryToken]
    public ActionResult Logout()
    {
        HttpContext.GetOwinContext()
                   .Authentication
                   .SignOut(DefaultAuthenticationTypes.ApplicationCookie);
        return RedirectToAction("Index", "Home");
    }
}
Tok prijave:
1. Korisnik šalje POST s email + password
2. SignInManager.PasswordSignInAsync provjerava hash lozinke u bazi
3. Na uspjehu: OWIN generira auth cookie (šifrirani JWT-like token)
4. Svaki sljedeći zahtjev: middleware čita cookie, kreira ClaimsPrincipal, popunjava User.Identity i User.IsInRole(...)
5. Na odjavi: cookie se briše iz preglednika

Korak 6: [Authorize] atribut — zaštita akcija

[Authorize] — svaki autentificirani korisnik može pristupiti
[Authorize(Roles = "Admin,SuperAdmin")] — samo navedene uloge
[AllowAnonymous] — pregazi [Authorize] na višem nivou (klasa → metoda)

Šta se desi neovlaštenom korisniku?
U ASP.NET MVC 5, [Authorize] uvijek preusmjerava na login stranicu — čak i ako je korisnik već prijavljen ali nema odgovarajuću ulogu. Ovo je poznano ograničenje MVC 5 — vidi Korak 9 (razlika s .NET Core gdje 401 vs 403 razlikuje "nije prijavljen" od "nema dozvolu").

Korak 7: UserManagementController

Nova datoteka: Controllers/UserManagementController.cs — dostupan samo SuperAdmin korisnicima 
[Authorize(Roles = "SuperAdmin")]
public class UserManagementController : BaseController
{
    // GET: UserManagement — lista svih korisnika s ulogama i odjelom
    public ActionResult Index()
    {
        using (var authDb = new ApplicationDbContext())
        using (var appDb = new StateStatisticsDBEntities())
        {
            var departments = appDb.Departments
                .ToDictionary(d => d.DepartmentID, d => d.DepartmentName);

            // Učitaj roleName lookup: roleId -> naziv uloge
            var allRoles = authDb.Roles.ToDictionary(r => r.Id, r => r.Name);

            var users = authDb.Users.Include("Roles").ToList();

            var list = users.Select(u => new UserListItem
            {
                Id = u.Id,
                FullName = u.FullName,
                Email = u.Email,
                // Pronađi nazive uloga kroz roleId u lookup-u
                Roles = u.Roles.Select(r => allRoles[r.RoleId]).ToList(),
                DepartmentName = u.DepartmentID.HasValue
                    ? departments.GetValueOrDefault(u.DepartmentID.Value) : null
            }).ToList();

            return View(list);
        }
    }

    // POST: UserManagement/Create — SuperAdmin kreira nalog
    [HttpPost, ValidateAntiForgeryToken]
    public async Task<ActionResult> Create(CreateUserViewModel model)
    {
        if (!ModelState.IsValid) { ... return View(model); }

        var user = new ApplicationUser
        {
            UserName = model.Email, Email = model.Email,
            FullName = model.FullName, DepartmentID = model.DepartmentID,
            EmailConfirmed = true
        };

        // UserManager.CreateAsync: validira email jedinstvenost, hashira lozinku, upisuje u bazu
        var result = await UserManager.CreateAsync(user, model.Password);
        if (result.Succeeded)
        {
            await UserManager.AddToRoleAsync(user.Id, model.Role);
            return RedirectToAction("Index");
        }
        // result.Errors sadrži tačne poruke (npr. "Password too short")
        foreach (var error in result.Errors)
            ModelState.AddModelError("", error);
        ...
    }

    // POST: UserManagement/Delete — ne može obrisati sam sebe ili zadnjeg SuperAdmin-a
    [HttpPost, ValidateAntiForgeryToken]
    public ActionResult Delete(string id)
    {
        if (id == User.Identity.GetUserId())
        { TempData["Error"] = "Ne možeš obrisati vlastiti nalog."; return ...; }

        using (var authDb = new ApplicationDbContext())
        {
            var superAdminRole = authDb.Roles.FirstOrDefault(r => r.Name == "SuperAdmin");
            int superAdminCount = authDb.Users
                .Count(u => u.Roles.Any(r => r.RoleId == superAdminRole.Id));
            bool isTarget = authDb.Users
                .Any(u => u.Id == id && u.Roles.Any(r => r.RoleId == superAdminRole.Id));

            if (isTarget && superAdminCount <= 1)
            { TempData["Error"] = "Zadnji SuperAdmin ne može biti obrisan."; return ...; }

            var user = authDb.Users.Find(id);
            user.Roles.Clear();  // Identity zahtijeva ručno brisanje relacija
            authDb.Users.Remove(user);
            authDb.SaveChanges();
        }
        return RedirectToAction("Index");
    }
}
Zašto user.Roles.Clear() prije brisanja?
Identity čuva UserRoles kao zasebnu tabelu (AspNetUserRoles) s foreign key vezama. EF neće automatski kaskadno brisati te redove — mora se eksplicitno ukloniti kolekcija relacija ili koristiti kaskadno brisanje na nivou FK-a (što Identity ne konfigurira automatski).

Korak 8: Departmentski-ograničen pristup (Editor uloga)

Zahtjev: Editor uloga može editovati Zaposlenike, Projekte i Izvještaje — ali samo one koji pripadaju istom odjelu kao Editor korisnik.
Nova datoteka: Infrastructure/BaseController.cs — svi kontroleri nasljeđuju ovaj umjesto Controller 
public class BaseController : Controller
{
    // UserManager iz OWIN contexta (lazy property)
    protected ApplicationUserManager UserManager
        => HttpContext.GetOwinContext().GetUserManager<ApplicationUserManager>();

    // Vraća true ako trenutni korisnik smije editovati zapis iz tog odjela.
    // SuperAdmin i Admin — uvijek smiju.
    // Editor — samo ako mu DepartmentID odgovara.
    protected bool CanEditDepartment(int? entityDepartmentId)
    {
        if (User.IsInRole("SuperAdmin") || User.IsInRole("Admin")) return true;
        if (!User.IsInRole("Editor")) return false;

        var userId = User.Identity.GetUserId();
        using (var db = new ApplicationDbContext())
        {
            var appUser = db.Users.Find(userId);
            return appUser != null && appUser.DepartmentID == entityDepartmentId;
        }
    }
}

U EmployeeController (i ProjectController, ReportController) koristimo:

[Authorize(Roles = "Admin,Editor,SuperAdmin")]
public ActionResult Edit(int id)
{
    using (var db = new StateStatisticsDBEntities())
    {
        var employee = db.Employees.Find(id);
        if (employee == null) return HttpNotFound();

        // Provjera departmentskog pristupa za Editor ulogu
        if (!CanEditDepartment(employee.DepartmentID))
            return new HttpStatusCodeResult(HttpStatusCode.Forbidden,
                "Nemaš pristup zaposlenicima iz ovog odjela.");

        PopulateDepartmentsSelectList(db, employee.DepartmentID);
        return View(employee);
    }
}

[HttpPost]
[Authorize(Roles = "Admin,Editor,SuperAdmin")]
[ValidateAntiForgeryToken]
public ActionResult Edit(Employee model)
{
    if (ModelState.IsValid)
    {
        using (var db = new StateStatisticsDBEntities())
        {
            // Provjera i na POST — ne oslanjaj se na GET provjeru jer korisnik
            // može poslati POST direktno (zaobilazeći GET formu)
            var existing = db.Employees.Find(model.EmployeeID);
            if (existing != null && !CanEditDepartment(existing.DepartmentID))
                return new HttpStatusCodeResult(HttpStatusCode.Forbidden,
                    "Nemaš pristup zaposlenicima iz ovog odjela.");

            model.ModifiedDate = DateTime.Now;
            db.Entry(model).State = EntityState.Modified;
            db.SaveChanges();
            return RedirectToAction("Index");
        }
    }
    ...
}
Zašto ponavljamo provjeru i na POST akciji?
Napadač može poslati HTTP POST zahtjev direktno (bez GET forme) sa proizvoljnim podacima. Ako provjeru radimo samo u GET akciji, to je sigurnosna rupa. Provjera mora biti i na POST — uvijek validiraj na serveru, nikad ne oslanjaj se samo na klijent.
Skrivanje dugmadi u View-ovima:
Kontroler vraća 403 ako korisnik nema pravo — ali za bolji UX, dugmad "Uredi" i "Obriši" skrivamo i u View-u pomoću User.IsInRole(): 
@Html.ActionLink("Detalji", "Details", new { id = item.EmployeeID }, ...)

@if (User.IsInRole("Admin") || User.IsInRole("SuperAdmin") || User.IsInRole("Editor"))
{
    @Html.ActionLink("Uredi", "Edit", new { id = item.EmployeeID }, ...)
}
@if (User.IsInRole("Admin") || User.IsInRole("SuperAdmin"))
{
    @Html.ActionLink("Obriši", "Delete", new { id = item.EmployeeID }, ...)
}
Skrivanje ≠ sigurnost. Skrivanje dugmadi u View-u je samo UX poboljšanje. Stvarna sigurnost se provodi na kontroleru ([Authorize] i CanEditDepartment). Nikad ne oslanjaj se isključivo na skrivanje HTML elemenata.

Korak 9: ASP.NET MVC 5 vs ASP.NET Core — razlike

Isti koncepti (uloge, cookie auth, Identity), ali drugačija implementacija:

Aspekt ASP.NET MVC 5 (.NET Framework) ASP.NET Core
Middleware OWIN (Startup.cs posebna klasa, [assembly: OwinStartup]) Ugrađen u framework (Program.cs, app.UseAuthentication())
DI kontejner Nema ugrađen DI — ručno kroz OWIN context (HttpContext.GetOwinContext().GetUserManager<T>()) Ugrađen DI (services.AddIdentity<...>()), injektira se kroz konstruktor
UserManager u kontroleru HttpContext.GetOwinContext().GetUserManager<ApplicationUserManager>() private readonly UserManager<ApplicationUser> _um;
public Controller(UserManager<ApplicationUser> um) { _um = um; }
Cookie auth konfiguracija app.UseCookieAuthentication(new CookieAuthenticationOptions { ... }) services.AddAuthentication().AddCookie(o => o.LoginPath = "/login")
[Authorize] sintaksa Identična Identična + Policy-based: [Authorize(Policy = "CanEdit")]
Policy-based auth Ne postoji (samo uloge ili custom atributi) services.AddAuthorization(o => o.AddPolicy("CanEdit", p => p.RequireRole("Admin").RequireClaim("dept")))
401 vs 403 Uvijek redirect na login (bez razlike) Razlikuje: 401 = nije prijavljen (redirect na login), 403 = nema dozvolu (forbidden page)
Password hasher PBKDF2 s SHA-1 (Identity v2) PBKDF2 s SHA-256 (Identity v3, kompatibilan s v2)
IdentityDbContext Isti koncept Isti koncept — ApplicationDbContext : IdentityDbContext<ApplicationUser>
Claims Dostupni, ali manje naglašeni Centralni — sve je ClaimsPrincipal, role su samo claims tipa RoleClaimType
[AllowAnonymous] Identičan Identičan
Migracije EF Migrations ili ručni SQL dotnet ef migrations add ... && dotnet ef database update
Seedovanje Ručno u Application_Start IHostedService ili app.SeedDatabase() u Program.cs
Kratak zaključak:
Koncepti su isti — autentifikacija, korisnici, uloge, [Authorize] atributi. Razlika je u infrastrukturi: MVC 5 koristi OWIN i ručno upravljanje dependency injection-om, dok .NET Core ima ugrađen DI kontejner i moderniji middleware pipeline. Prelaz s MVC 5 na Core zahtijeva refaktor bootstrap koda, ali logika kontrolera (role provjere, CanEditDepartment) ostaje gotovo identična.
